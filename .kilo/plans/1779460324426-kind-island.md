# User Feedback Sprint Plan

_Previous content (test suite) was implemented successfully (77 passing tests). This plan covers the user-feedback sprint._

---

## Sprint Goals (from feedback report)

1. **Label inconsistency (Critical)** — Table shows "Certificado" (DGP status) and the report shows "Não Certificado" (validation result). Both can be correct simultaneously; the user is confused because the two concepts are not labelled distinctly. Fix: rename + add clarifying labels (user's preferred option).

2. **Bancas counter = 0** — `data-groups.json` / `data.json` have no `bancas` key in `producoes` because the SUAP scraper hasn't extracted the "Participação em Bancas" sheet yet. The data is **not available yet** (confirmed by user). Action: add a visible notice in the UI when bancas data is absent, and document the gap clearly in the report.

3. **Mobile responsiveness & performance** — The 7-column table forces `min-width: 720px` on mobile, breaking layout. No touch-target tuning exists. Fix: CSS/media-query improvements and a card/accordion view for narrow screens.

4. **Individual researcher filter** — New feature: allow filtering/searching by a single researcher's name to support data auditing.

## Detailed Tasks

### 1 — Label Inconsistency (Critical)

**Root cause:** `CNPQ-01` checks `g.Situacao === 'Certificado'` (exact match). A group can be `"Certificado"` in DGP but still receive `NAO_CERTIFICADO` as the IFBA validation parecer when it fails other criteria. The table column header is unlabelled as to its source; the report footer label "Não Certificado" is ambiguous.

**Changes:**
- `index.html`: rename column header "Situação" → "Situação DGP" (`<th data-sort="situacao" ...>Situação DGP</th>`).
- `index.html`: add a tooltip/`title` on that `<th>` explaining it reflects the DGP registry, not the PRPGI validation.
- `criterios.js` / `script.js` (`PARECER_LABELS_UPPER`): rename the label for `NAO_CERTIFICADO` from "NÃO CERTIFICADO" to "NÃO CONFORME" and `PENDENTE` → "CERTIFICADO c/ PENDÊNCIAS" (the current value is already "CERTIFICADO c/ ADVERTÊNCIAS" — keep as-is or refine per feedback phrasing "Necessita de ajuste"; user chose "Necessita de ajuste" / "Não conforme" → use "NÃO CONFORME").
- `script.js` (`gerarRelatorio`): in the generated report HTML, label section "7. Parecer Final" subtitle text to read "Parecer de Validação PRPGI/IFBA" and add a note under the badge: "Nota: a Situação DGP do grupo é '{resultado.grupo.situacao}', que reflete o cadastro na Plataforma Lattes/CNPq, independente deste parecer de validação institucional."
- `script.js` (`renderTableRows`): add `title` attribute to the status badge cell clarifying "Situação no DGP/CNPq".

### 2 — Bancas Counter = 0 (Data Gap)

**Root cause:** The SUAP scraper has not yet exported a "Participação em Bancas" sheet. The `producoes.bancas` array is absent from both JSON files, so `ValidadorGrupo` scores zero for bancas.

**Changes (UI-level, no data pipeline):**
- `criterios.js`: in `_getProducaoPorCategoria`, detect when `prods.bancas` is `undefined` (as opposed to an empty array `[]`) and expose that state. Add a `bancasIndisponivel` flag to the result of `getProducaoPorCategoria` (or return `{items: [], indisponivel: true}` for the bancas key).
- `script.js` (`gerarRelatorio`): when `bancasIndisponivel` is true, render a row in the metrics grid saying "Bancas: Dado não disponível na fonte atual".
- `script.js` (the validation panel `renderChecklistHTML`): the bancas scoring criterion already shows 0; add a footnote `†` explaining bancas data is pending extraction from SUAP.
- Add a visible `showToast` warning after data loads if `bancas` key is absent: "Atenção: dados de participação em bancas ainda não disponíveis — o indicador será zero até atualização da base."

### 3 — Mobile Responsiveness & Performance

**Root cause (layout):** `#groups-table { min-width: 720px }` inside `@media (max-width: 766px)` forces the table wider than the viewport. The table-wrapper uses `overflow-x: auto` so it scrolls, but the rest of the page stacks OK. More UX issues: period buttons can overflow on small screens, and touch targets (group table rows) are small.

**Changes (`style.css`):**
- Remove `min-width: 720px` from the mobile breakpoint. Instead, allow the table to be responsive by hiding lower-priority columns (Area, Ano) on very narrow screens using `display: none` on those `<td>`/`<th>` via a CSS class.
- Add `@media (max-width: 480px)` block to hide the "Área" and "Ano" columns (they are lower priority than Nome, Campus, Situação, Pesquisadores).
- Increase row `min-height` / `padding` for touch targets on mobile: bump `td` padding to `0.75rem 0.6rem` on mobile.
- Ensure `.period-btn-group` wraps gracefully (already `flex-wrap: wrap`) and reduce button `padding` on small screens.
- Add `font-size: 0.82rem` to `#groups-table` on mobile for better text fit.

**Changes (`index.html`):**
- Add `data-mobile-hidden` attribute to the `<th>` / `<td>` for "Área" and "Ano" columns so CSS can target them cleanly.

**Performance note:** the slowness reported on mobile is likely network (fetching `data.json` + `data-groups.json` from GitHub Pages). No JavaScript profiling is possible here, but we can add a `<link rel="preconnect">` hint in `index.html` for the GitHub Pages origin, and ensure the loading overlay stays visible until data is ready (already implemented).

### 4 — Individual Researcher Filter (New Feature)

**Scope:** Add a search/filter input in the landing section that filters the groups table to only show groups whose `PesquisadoresNomes` (or `membrosMap`) contains the searched researcher name.

**Changes (`index.html`):**
- Add a new `<input type="text" id="researcher-search" placeholder="Buscar por pesquisador...">` in the `.landing-controls` block, after the existing `#group-search`.
- Add a `<label>` with `class="sr-only"` for accessibility.

**Changes (`style.css`):**
- `#researcher-search` — style identically to `#group-search` (same class or duplicate rule).
- Include in the mobile stacking rules.

**Changes (`script.js`):**
- In `filterGroupsTable`: read the new `#researcher-search` value; add a `matchesResearcher` check that searches `(g.PesquisadoresNomes || '').toLowerCase()` and also searches `membrosMap` names if available.
- In `filterCurrentTableRows`: same logic.
- Wire up the `input` event listener on `DOMContentLoaded` (same pattern as `groupSearch`).
